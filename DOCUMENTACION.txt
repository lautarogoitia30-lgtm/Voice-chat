================================================================================
                     1. INTRODUCCIÓN AL PROYECTO
================================================================================

Voice-Chat es una aplicación de chat de voz y texto similar a Discord o 
TeamSpeak. Permite:

✓ Crear grupos (servidores)
✓ Crear canales de texto y voz dentro de los grupos
✓ Chatear en tiempo real por texto
✓ Chatear en tiempo real por voz (usando LiveKit)
✓ Invitar usuarios a grupos

================================================================================
                     2. ARQUITECTURA GENERAL
================================================================================

El proyecto tiene una arquitectura cliente-servidor típica:

    ┌─────────────────┐          ┌─────────────────┐
    │                 │          │                 │
    │   FRONTEND      │◄────────►│   BACKEND       │
    │   (Navegador)  │   HTTP   │   (FastAPI)     │
    │                 │          │                 │
    └─────────────────┘          └─────────────────┘
            │                            │
            │ WebSocket                  │ SQLite
            │                            │
            ▼                            ▼
    ┌─────────────────┐          ┌─────────────────┐
    │  Chat de Texto  │          │  Base de Datos  │
    │  (tiempo real) │          │   (SQLite)      │
    └─────────────────┘          └─────────────────┘
                                        │
                                        ▼
                               ┌─────────────────┐
                               │   LiveKit       │
                               │  (Voz/音频)     │
                               └─────────────────┘

FLUJO DE DATOS:

1. Usuario se registra/login → Backend verifica y devuelve JWT token
2. Frontend guarda el token en localStorage
3. Para chatear por texto → WebSocket conecta directamente
4. Para chatear por voz → Backend genera token de LiveKit y conecta

================================================================================
                     5. BASE DE DATOS
================================================================================

La base de datos tiene las siguientes tablas:

TABLA: users
├── id (INTEGER PRIMARY KEY)      - ID único del usuario
├── username (TEXT)               - Nombre de usuario
├── email (TEXT)                  - Email del usuario
├── password_hash (TEXT)          - Contraseña encriptada
└── created_at (DATETIME)        - Fecha de creación

TABLA: groups
├── id (INTEGER PRIMARY KEY)      - ID único del grupo
├── name (TEXT)                   - Nombre del grupo
├── owner_id (INTEGER)            - ID del usuario que creó el grupo
└── created_at (DATETIME)        - Fecha de creación

TABLA: channels
├── id (INTEGER PRIMARY KEY)      - ID único del canal
├── group_id (INTEGER)           - ID del grupo al que pertenece
├── name (TEXT)                  - Nombre del canal
├── type (TEXT)                  - "text" o "voice"
└── created_at (DATETIME)       - Fecha de creación

TABLA: group_members
├── id (INTEGER PRIMARY KEY)
├── group_id (INTEGER)           - ID del grupo
├── user_id (INTEGER)            - ID del usuario
└── joined_at (DATETIME)         - Fecha en que se unió

TABLA: voice_participants
├── id (INTEGER PRIMARY KEY)
├── channel_id (INTEGER)         - ID del canal de voz
├── user_id (INTEGER)            - ID del usuario
└── joined_at (DATETIME)         - Fecha en que se unió al voz

RELACIONES ENTRE TABLAS:

users (1) ──────< (N) group_members
users (1) ──────< (N) groups (como owner)
groups (1) ─────< (N) channels
groups (1) ─────< (N) group_members
channels (1) ──< (N) voice_participants

6.1 MODELOS (backend/models.py)
-----------------------------------------------------------------

Los modelos definen la estructura de las tablas en la base de datos.

    class User(Base):
        - id: Integer primary key
        - username: String(50) unique - NO puede haber dos usuarios con mismo nombre
        - email: String(100) unique - NO puede haber dos usuarios con mismo email
        - password_hash: String - Contraseña encriptada con bcrypt
        - created_at: DateTime - Cuándo se creó el usuario

    class Group(Base):
        - id: Integer primary key
        - name: String(100) - Nombre del grupo
        - owner_id: Integer FK - Quién creó el grupo
        - owner: Relationship - Relación con User
        - created_at: DateTime

    class Channel(Base):
        - id: Integer primary key
        - group_id: Integer FK - A qué grupo pertenece
        - name: String(100) - Nombre del canal
        - type: Enum - "text" o "voice"
        - created_at: DateTime

    class GroupMember(Base):
        - id: Integer primary key
        - group_id: Integer FK
        - user_id: Integer FK
        - joined_at: DateTime

    class VoiceParticipant(Base):
        - id: Integer primary key
        - channel_id: Integer FK - Canal de voz donde está
        - user_id: Integer FK - Usuario que está en voz
        - joined_at: DateTime

6.2 AUTENTICACIÓN (backend/auth.py)
-----------------------------------------------------------------

Este archivo contiene las funciones de seguridad:

A) hash_password(password)
   - Recibe: "micontraseña"
   - Retorna: "$2b$12$..." (hash de bcrypt)
   - Usa: bcrypt.gensalt() y bcrypt.hashpw()

B) verify_password(plain_password, hashed_password)
   - Recibe: contraseña plana y hash guardado
   - Retorna: True si coinciden, False si no
   - Usa: bcrypt.checkpw()

C) create_jwt_token(user_id, username)
   - Recibe: user_id=1, username="Lauti"
   - Retorna: "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
   - El JWT contiene: sub (user_id), username, exp (expiración)
   - Expira en 1440 minutos (24 horas) por defecto

D) verify_jwt_token(token)
   - Recibe: el token JWT
   - Retorna: {"user_id": 1, "username": "Lauti", "exp": ...}
   - Lanza excepción si el token está expirado o es inválido

E) get_current_user(credentials)
   - Es una dependencia de FastAPI
   - Se usa en las rutas que requieren autenticación
   - Lee el header "Authorization: Bearer <token>"
   - Retorna el diccionario con user_id y username

6.3 RUTAS DE AUTENTICACIÓN (backend/routes/auth.py)
-----------------------------------------------------------------

Son los endpoints para registro y login:

POST /auth/register
- Input: {"username": "...", "email": "...", "password": "..."}
- Proceso:
  1. Verifica que el username no exista
  2. Verifica que el email no exista
  3. Hashea la contraseña
  4. Crea el usuario en la DB
- Output: {"id": 1, "username": "Lauti", "email": "lauti@test.com"}

POST /auth/login
- Input: {"username": "Lauti", "password": "password123"}
- Proceso:
  1. Busca el usuario por username
  2. Verifica la contraseña con verify_password
  3. Crea un JWT token con create_jwt_token
- Output: {
    "access_token": "eyJhbGci...",
    "token_type": "bearer",
    "user_id": 1,
    "username": "Lauti"
  }

-----------------------------------------------------------------
6.4 RUTAS DE GRUPOS (backend/routes/groups.py)
-----------------------------------------------------------------

Endpoints para gestionar grupos:

GET /groups
- Requiere: JWT token
- Proceso: Busca todos los grupos donde el usuario es miembro
- Output: [{"id": 1, "name": "Hola", "owner_id": 1}, ...]

POST /groups
- Requiere: JWT token
- Input: {"name": "Mi Grupo"}
- Proceso:
  1. Crea el grupo con el usuario actual como owner
  2. Agrega al owner como miembro automáticamente
- Output: {"id": 2, "name": "Mi Grupo", "owner_id": 1}

GET /groups/{group_id}
- Requiere: JWT token
- Proceso: Devuelve los detalles del grupo
- Output: {"id": 1, "name": "Hola", "owner_id": 1, "member_count": 2}

POST /groups/{group_id}/invite
- Requiere: JWT token
- Input: {"username": "MrPorteno"}
- Proceso:
  1. Busca el usuario a invitar
  2. Verifica que el actual sea miembro del grupo
  3. Agrega el usuario a group_members
- Output: {"status": "ok", "message": "User MrPorteno added to group"}

GET /groups/{group_id}/members
- Requiere: JWT token
- Proceso: Lista todos los miembros del grupo
- Output: [{"id": 1, "username": "Lauti", "email": "..."}, ...]

-----------------------------------------------------------------
6.5 RUTAS DE CANALES (backend/routes/channels.py)
-----------------------------------------------------------------

Endpoints para gestionar canales de texto y voz:

GET /channels/groups/{group_id}/channels
- Requiere: JWT token
- Proceso: Lista todos los canales de un grupo
- Output: [{"id": 1, "group_id": 1, "name": "General", "type": "voice"}, ...]

POST /channels/groups/{group_id}/channels
- Requiere: JWT token
- Input: {"name": "General", "type": "voice"}
- Proceso: Crea un nuevo canal en el grupo
- Output: {"id": 2, "group_id": 1, "name": "General", "type": "voice"}

POST /channels/{channel_id}/voice/join
- Requiere: JWT token
- Proceso:
  1. Verifica que el canal sea de tipo "voice"
  2. Verifica que el usuario sea miembro del grupo
  3. Agrega el usuario a voice_participants
- Output: {"status": "ok", "message": "Joined voice channel"}

POST /channels/{channel_id}/voice/leave
- Requiere: JWT token
- Proceso: Elimina el usuario de voice_participants
- Output: {"status": "ok", "message": "Left voice channel"}

GET /channels/{channel_id}/voice/participants
- Requiere: JWT token
- Proceso: Lista quién está actualmente en el canal de voz
- Output: [{"user_id": 1, "username": "Lauti", "joined_at": "..."}, ...]

-----------------------------------------------------------------
6.6 RUTAS DE LIVEKIT (backend/routes/livekit.py)
-----------------------------------------------------------------

Endpoint para generar el token de voz:

POST /livekit/token
- Requiere: JWT token
- Input: {"channel_id": 1}
- Proceso:
  1. Verifica que el canal exista y sea de voz
  2. Verifica que el usuario sea miembro del grupo
  3. Genera un token de LiveKit con:
     - identity = str(user_id)  → "1" o "2"
     - name = username           → "Lauti" o "MrPorteno"
     - room = f"channel-{channel_id}" → "channel-1"
- Output: {
    "token": "eyJhbGci...",
    "url": "wss://...",
    "room_name": "channel-1"
  }

El token permite al frontend conectarse a LiveKit como ese usuario
específico. El "identity" es lo que usa LiveKit para identificar a
cada participante en la sala.

6.7 WEBSOCKET (backend/websocket.py)

7.4 LIVEKIT CLIENT (frontend/js/livekit.js)
-----------------------------------------------------------------

Maneja la conexión de voz usando LiveKit:

MÉTODOS:

A) livekitClient.connect(url, token)
   - url: "wss://..." (del token de /livekit/token)
   - token: "eyJ..." (del token de /livekit/token)
   - Proceso:
     1. Crea una Room de LiveKit
     2. Configura eventos:
        - ConnectionStateChanged
        - ParticipantConnected
        - ParticipantDisconnected
        - TrackSubscribed (audio recibido)
        - TrackUnsubscribed
        - ActiveSpeakersChanged
     3. Conecta al room
     4. Guarda localParticipant

B) livekitClient.publishMicrophone()
   - Solicita acceso al micrófono del navegador
   - Publica el track de audio en el room
   - Usa setMicrophoneEnabled(true) de LiveKit v2

C) livekitClient.getParticipants()
   - Retorna array de participantes remotos
   - Cada participante tiene: identity, name, isSpeaking

D) livekitClient.setMuted(muted)
   - muted = true: silencia el micrófono
   - muted = false: activa el micrófono

E) livekitClient.setDeafened(deafened)
   - deafened = true: silenciar speakers
   - deafened = false: activar speakers

F) livekitClient.disconnect()
   - Desconecta del room de LiveKit

EVENTOS IMPORTANTES:

- TrackSubscribed: Se dispara cuando alguien publica audio
  - El código hace: track.attach() para reproducir el audio
  - Crea un elemento <audio> y lo agrega al DOM

- ParticipantConnected/Disconnected: Notifica cuando alguien entra/sale

================================================================================
                     8. CÓMO FUNCIONA EL CHAT DE VOZ (LIVEKIT)
================================================================================

ENTENDIENDO LIVEKIT:

LiveKit es un servicio de WebRTC que maneja la transmisión de audio/video
en tiempo real. Nosotros usamos solo audio (voz).

CONCEPTOS CLAVE:

A) ROOM (Sala)
   - Es donde los participantes se conectan
   - Cada canal de voz tiene su propio room
   - Nombre del room: "channel-{channel_id}" ej: "channel-1"
   - URL del room: wss://... (proporcionado por LiveKit)

B) PARTICIPANT (Participante)
   - Cada usuario que se conecta a un room
   - Tiene un identity único (su user_id como string: "1", "2")
   - Tiene un name (su username: "Lauti", "MrPorteno")

C) TRACK (Pista)
   - Audio o video publicado por un participante
   - Nosotros usamos solo audio
   - Cuando publicás tu micrófono, creás un audio track
   - Cuando recibís audio de otro, lo llamás "subscribed"

FLUJO COMPLETO DE VOZ:

1. Usuario hace click en "Unirse"

2. Frontend llama: API.channels.joinVoice(channelId)
   - Backend: Agrega usuario a voice_participants en DB

3. Frontend llama: API.livekit.getToken(channelId)
   - Backend: Genera token JWT con:
     * identity = "1" (el user_id)
     * name = "Lauti"
     * room = "channel-1"
   - Retorna: {token, url, room_name}

4. Frontend llama: livekitClient.connect(url, token)
   - LiveKit: Valida el token
   - LiveKit: Crea la conexión WebRTC
   - LiveKit: Emite evento "ParticipantActive" para el usuario local

5. Frontend llama: livekitClient.publishMicrophone()
   - Navegador: Solicita acceso al micrófono
   - LiveKit: Recibe el audio y lo publica en el room
   - Otros participantes: Reciben evento "TrackSubscribed"

6. Cuando otro usuario se conecta:
   - Tu frontend recibe: "ParticipantConnected"
   - Tu código lo agrega a la lista de participantes

7. Cuando alguien habla:
   - LiveKit detecta quién tiene audio activo
   - Emite: "ActiveSpeakersChanged"
   - Tu código actualiza la UI para mostrar quién habla

8. Cuando alguien se va:
   - LiveKit emite: "ParticipantDisconnected"
   - Tu código lo remueve de la lista

9. Para escuchar audio:
   - Cuando llega un TrackSubscribed con kind="audio"
   - Tu código hace: const audioElement = track.attach()
   - Esto crea un <audio> HTML que reproduce el audio

================================================================================
                     9. CÓMO FUNCIONA EL CHAT DE TEXTO (WEBSOCKET)
================================================================================

El chat de texto usa WebSocket, una conexión bidireccional permanente
entre el navegador y el servidor.

FLUJO:

1. Usuario entra a un canal de texto

2. Frontend: wsClient.connect(channelId)
   - Conecta a: ws://localhost:8001/ws/chat/1
   - El backend acepta la conexión

3. Usuario escribe un mensaje y presiona "Enviar"

4. Frontend: wsClient.send(JSON.stringify(mensaje))
   - Envía: {
       channel_id: 1,
       sender_id: 1,
       sender_username: "Lauti",
       content: "Hola!",
       timestamp: "2024-01-01T12:00:00"
     }

5. Backend (websocket.py):
   - Recibe el mensaje
   - Lo reenvía a TODOS los WebSockets conectados a ese canal
   - (excepto el que lo envió, o a todos según la implementación)

6. Frontend (onMessage):
   - Recibe el mensaje JSON
   - Lo parsea
   - Llama handleWebSocketMessage(message)
   - Crea un elemento DOM y lo agrega a messagesList

SOLUCIÓN: Usar otro puerto: python -m uvicorn ... --port 8001

CAUSA: El JWT expiró (24 horas por defecto)

SOLUCIÓN: Hacer login de nuevo para obtener nuevo token

PROBLEMA: "Cannot connect to LiveKit"

CAUSA: Variables de entorno de LiveKit no configuradas

SOLUCIÓN: Configurar LIVEKIT_URL, LIVEKIT_API_KEY, LIVEKIT_API_SECRET
